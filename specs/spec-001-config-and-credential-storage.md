## Summary

resume-roast will call LLM APIs and therefore needs an API key, plus a home for
non-secret user settings. This spec adds a generic `persistence/` package for
JSON-backed stores — more stores will follow, so the layout is domain-scoped
from the start: each domain is a package composed of **dataclasses**, a
**parser** (loaded JSON → dataclass), and a **store class** (the loader and
public interface, which uses the parser), with shared JSON-loading errors at
the persistence root. The first two domains are `config_store/`
(`~/.resume-roast/config.json`, non-secret settings) and `credentials_store/`
(`~/.resume-roast/credentials.json`, secrets) — split so credentials can later
be encrypted without touching config handling. The spec also adds the CLI
skeleton (the `resume-roast` entry point does not exist yet) with a `config`
subcommand group: `resume-roast config credentials` prompts for the Anthropic
API key with hidden input and saves it, confirming with a masked echo. Both
files are small, human-readable JSON.

### Existing Code

- `src/resume_roast/__init__.py` — empty package; the only production file.
- `pyproject.toml` — has **no** `[project.scripts]` entry yet;
  `docs/development.md` documents running `resume-roast`, which this spec makes

- **Explicit non-goals** (required):
  - **No encryption** of credentials — the config/credentials separation exists
    to make encryption a clean later spec.
  - **No `config models` subcommand** — later spec; the `config` group and
    `config_store/` must allow adding it without rework.
  - **No consumption of the API key** — nothing calls any API in this spec.

## Invariants Referenced

- [INV-001: Secrets Stay in the Credential Store](../docs/invariants/inv-001-secrets-stay-in-credential-store.md)
  — the API key must never appear unmasked in CLI output or in `config.json`;
  enforced by `test_config_credentials_masks_key_in_output` and
  `test_saving_credentials_never_touches_config_file` below.
- [INV-002: JSON Never Crosses a Boundary Untyped](../docs/invariants/inv-002-json-never-crosses-a-boundary-untyped.md)
  — this spec establishes the parser seam the invariant requires: every store
  loads raw JSON only inside its `parser.py`, and `load()`/`save()` return and
  accept typed dataclasses, never `dict[str, Any]`; enforced by
  `test_load_raises_invalid_schema_error` (both domains) and by
  `pyright --strict` on the store/parser signatures.

## Pre-implementation Self-Check

- DP-001 — CLI tests run through Typer's `CliRunner`; store tests assert JSON
  on disk and returned dataclasses. No mocks of internals; the only boundary
  faked is the storage directory (a `tmp_path`, injected via the store
  constructor / `RESUME_ROAST_HOME`). Parsers not tested directly.
- DP-002 — `store_dir` fixture shared via `tests/conftest.py`; masking and
  invalid-schema cases parametrized.
- INV-001 — guaranteed via the two named scenarios; masking centralized in
  `mask_secret`; error messages never include file contents.
- INV-002 — guaranteed structurally: `json_file.read_json_object` returns
  `dict[str, Any]` and is called only from `parser.py` modules; every store's
  `load()` return type and every test's assertions are against the domain
  dataclass, never the raw dict. No test hand-builds a `Credentials`/`Config`
  from a dict, bypassing the parser.

## Test Plan (written first)

### `tests/persistence/test_credentials_store.py`

- `test_save_then_load_roundtrips_key` — Given a store dir, when credentials
  are saved and loaded via `CredentialsStore`, the same key comes back.
- `test_load_returns_none_when_file_missing` — Given an empty store dir,
  `load()` returns `None` (first run).
- `test_save_creates_store_dir` — Given a nonexistent store dir path, `save()`
  creates it.
- `test_credentials_file_is_owner_only_on_posix` — Given a POSIX platform,
  after `save()`, `credentials.json` has mode `0600` (skipped on Windows,
  where `chmod` is best-effort).
- `test_load_raises_invalid_json_error_on_corrupt_file` — Given a
  `credentials.json` containing invalid JSON, `load()` raises
  `InvalidJsonError` whose message names the file path but not its contents
  (ties to INV-001).
- `test_load_raises_invalid_schema_error` — Parametrized over: valid JSON
  missing `anthropic_api_key`; blank key value; top-level JSON that is not an
  object. Each raises `InvalidSchemaError`, and each message names the path
  but never embeds the offending value (ties to INV-001 and, by rejecting
  every malformed shape before a model is constructed, INV-002).
- `test_saving_credentials_never_touches_config_file` — Given an existing
  `config.json`, saving credentials leaves its bytes unchanged and never
  writes the key into it (enforces INV-001).
- `test_save_leaves_no_temp_artifacts` — After `save()`, the store dir
  contains only the expected file(s) (atomic-write cleanup).
- `test_mask_secret_shows_at_most_last_four` — Parametrized: long key →
  `****` + last 4; key of 4 or fewer chars → `"****"` (enforces INV-001).

## Changes Required

### `src/resume_roast/persistence/errors.py`

- **Interface**:

  ```python
  class PersistenceError(Exception):
      """Base for all persistence failures."""

  class InvalidJsonError(PersistenceError):
      """A store file exists but is not valid JSON."""

  class InvalidSchemaError(PersistenceError):
      """A store file is valid JSON but not the shape the parser expects."""
  ```

- **Behavior**: pure exception definitions, shared by every store domain.
  Messages must name the offending file path and the problem; they must never
  embed file contents (INV-001 — the credentials file's contents are secret).
- **Acceptance Examples**: exercised through the store examples below.
- **Data flow**: `InvalidJsonError` is raised by `json_file.py`;
  `InvalidSchemaError` by `json_file.py` (non-object top level — part of its
  return-a-dict contract) and by domain parsers (domain shape);
  `PersistenceError` directly for wrapped environmental write failures. All
  are caught only at the CLI boundary.
- **Edge cases**: none of their own.
- **Strategy**: stdlib only. Avoid: per-domain error subclasses until a caller
  needs to distinguish domains.
- **Tests**: covered transitively via store scenarios in *Test Plan*.

### `src/resume_roast/persistence/json_file.py`

- **Interface**:

  ```python
  def read_json_object(path: Path) -> dict[str, Any]: ...
  def write_json_object(path: Path, data: dict[str, Any], *, secure: bool = False) -> None: ...
  ```

- **Behavior**: `read_json_object` parses the file at `path`; raises
  `InvalidJsonError` (naming the path, not the contents) on parse failure and
  `InvalidSchemaError` if the top-level value is not a JSON object. Missing
  files are the *store's* concern — callers check existence first;
  `FileNotFoundError` propagates as-is. `write_json_object` creates the parent
  directory if missing (mode `0700` best-effort), serializes with `indent=2`,
  writes atomically (temp file in the same directory, then `os.replace`), and
  with `secure=True` sets file mode `0600` best-effort (no-op semantics on
  Windows are acceptable). Environmental write failures (`OSError` from
  mkdir/write/replace) are wrapped in `PersistenceError` naming the path, so
  the CLI's single `except PersistenceError` covers them.
- **Acceptance Examples**: exercised through the store examples below.
- **Data flow**: the only module that touches `json` or the filesystem; every
  store class calls it.
- **Edge cases**: parent dir missing; temp-file cleanup on success (no
  artifacts left); Windows `chmod` best-effort; concurrent writers out of
  scope (single-user CLI) but atomic replace prevents torn files.
- **Strategy**: stdlib only — `json`, `pathlib`, `tempfile`, `os`. Avoid:
  reading env vars, printing, importing domain modules.
- **Tests**: covered transitively via store scenarios (corrupt file, dir
  creation, permissions, no-temp-artifacts) in *Test Plan*.

### `src/resume_roast/persistence/paths.py`

- **Interface**:

  ```python
  def storage_dir() -> Path: ...
  ```

- **Behavior**: returns `Path(os.environ["RESUME_ROAST_HOME"])` when the
  variable is set and non-empty, else `Path.home() / ".resume-roast"`. Reads
  the env var at call time, not import time.
- **Acceptance Examples**:

  ```text
  Input:  storage_dir()   # RESUME_ROAST_HOME=/tmp/rr set
  Output: Path("/tmp/rr")
  ```

  ```text
  Input:  storage_dir()   # RESUME_ROAST_HOME unset
  Output: Path.home() / ".resume-roast"
  ```

- **Data flow**: called by the CLI to construct stores; stores themselves take
  an explicit `base_dir` and never read the environment.
- **Edge cases**: empty-string env value treated as unset.
- **Strategy**: stdlib only. Avoid: caching the result at module level.
- **Tests**: `tests/persistence/test_paths.py` scenarios in *Test Plan*.

### `src/resume_roast/persistence/credentials_store/` (models, parser, store)

- **Interface**:

  ```python
  # models.py
  @dataclass(frozen=True)
  class Credentials:
      anthropic_api_key: str

  def mask_secret(value: str) -> str: ...

  # parser.py
  def parse_credentials(data: dict[str, Any]) -> Credentials: ...

  # store.py
  class CredentialsStore:
      FILENAME = "credentials.json"

      def __init__(self, base_dir: Path) -> None: ...
      @property
      def path(self) -> Path: ...
      def load(self) -> Credentials | None: ...
      def save(self, credentials: Credentials) -> None: ...
  ```

- **Behavior**: `parse_credentials` raises `InvalidSchemaError` when
  `anthropic_api_key` is missing, not a string, or blank after stripping.
  `CredentialsStore.load` returns `None` when the file does not exist,
  otherwise `read_json_object` → `parse_credentials`. `save` writes
  `{"anthropic_api_key": <key>}` via `write_json_object(..., secure=True)`.
  `mask_secret` returns `"****"` + the last 4 characters, or `"****"` alone
  when the value has 4 or fewer characters.
- **Acceptance Examples**:

  ```text
  Input:  CredentialsStore(dir).save(Credentials("sk-ant-key-9876")); CredentialsStore(dir).load()
  Output: Credentials(anthropic_api_key="sk-ant-key-9876")
  ```

  ```text
  Input:  CredentialsStore(dir).load()   # no credentials.json present
  Output: None
  ```

  ```text
  Input:  CredentialsStore(dir).load()   # credentials.json contains "not json{"
  Output: raises InvalidJsonError naming <dir>/credentials.json
  ```

  ```text
  Input:  CredentialsStore(dir).load()   # file contains {"anthropic_api_key": "  "}
  Output: raises InvalidSchemaError
  ```

  ```text
  Input:  mask_secret("sk-ant-key-9876")
  Output: "****9876"
  ```

  ```text
  Input:  mask_secret("abc")
  Output: "****"
  ```

- **Data flow**: store → `json_file` primitives → parser → dataclass. The
  package `__init__.py` re-exports `Credentials`, `CredentialsStore`, and
  `mask_secret`; callers import from the package, not submodules.
- **Edge cases**: first run (no file → `None`); blank/whitespace key in file;
  non-object top level; overwrite of an existing file (atomic).
- **Strategy**: stdlib only. The parser owns all domain schema knowledge; the
  store owns file lifecycle; models own masking. Avoid: the store touching `json`
  directly, logging or printing secret values anywhere (INV-001).
- **Tests**: `tests/persistence/test_credentials_store.py` scenarios in *Test
  Plan* — all through `CredentialsStore` and `mask_secret`; the parser is
  covered transitively (DP-001).

### `src/resume_roast/persistence/config_store/` (models, parser, store)

- **Interface**:

  ```python
  # models.py
  @dataclass(frozen=True)
  class Config:
      """Intentionally empty in SPEC-001; fields arrive with later specs."""

  # parser.py
  def parse_config(data: dict[str, Any]) -> Config: ...

  # store.py
  class ConfigStore:
      FILENAME = "config.json"

      def __init__(self, base_dir: Path) -> None: ...
      @property
      def path(self) -> Path: ...
      def load(self) -> Config: ...
      def save(self, config: Config) -> None: ...
  ```

- **Behavior**: `parse_config` accepts any JSON object (unknown keys ignored —
  forward compatibility) and returns `Config()`. `ConfigStore.load` returns
  `Config()` when the file does not exist (implicit default — config is
  optional, credentials are not, hence the asymmetry with
  `CredentialsStore.load`). `save` writes `{}` via `write_json_object`
  (not secure — nothing secret).
- **Acceptance Examples**:

  ```text
  Input:  ConfigStore(dir).save(Config()); ConfigStore(dir).load()
  Output: Config()   # config.json on disk contains {}
  ```

  ```text
  Input:  ConfigStore(dir).load()   # no config.json present
  Output: Config()
  ```

  ```text
  Input:  ConfigStore(dir).load()   # config.json contains [1, 2]
  Output: raises InvalidSchemaError
  ```

- **Data flow / Edge cases / Strategy**: mirror `credentials_store/`; unknown
  keys tolerated by the parser.
- **Tests**: `tests/persistence/test_config_store.py` scenarios in *Test Plan*.

### `src/resume_roast/cli.py`

- **Interface**:

  ```python
  app = typer.Typer(no_args_is_help=True)
  config_app = typer.Typer(no_args_is_help=True)
  app.add_typer(config_app, name="config", help="Manage settings and credentials.")

  @config_app.command("credentials")
  def credentials() -> None: ...
  ```

- **Behavior**: `credentials` prompts "Anthropic API key" with
  `hide_input=True, confirmation_prompt=True`; strips whitespace; a blank
  result prints an error to stderr and exits with code 1 without writing;
  otherwise saves via `CredentialsStore(storage_dir()).save(...)` and prints
  `Saved Anthropic API key ****<last4> to <path>` (exit 0). A
  `PersistenceError` is caught and reported as a one-line error (exit 1),
  never a traceback.
- **Acceptance Examples**:

  ```text
  Input:  resume-roast config credentials   (typing sk-ant-key-9876 twice, hidden)
  Output: Saved Anthropic API key ****9876 to ~/.resume-roast/credentials.json ; exit 0
  ```

  ```text
  Input:  resume-roast config credentials   (typing only whitespace, twice)
  Output: Error: API key cannot be empty ; exit 1 ; no credentials.json written
  ```

  ```text
  Input:  resume-roast config
  Output: group help listing the credentials subcommand
  ```

- **Data flow**: Typer prompt → strip/validate → `Credentials` →
  `CredentialsStore.save` → masked confirmation via `mask_secret`. The CLI is
  the only module that prints and the only module that calls `storage_dir()`.
- **Edge cases**: mismatched confirmation input and truly-empty input (the
  prompt library re-prompts natively for both — accepted behavior, not custom
  code; the CLI's blank check therefore fires only on whitespace-that-strips-
  to-empty); existing key silently overwritten (documented behavior);
  unwritable store dir surfaces as a `PersistenceError` → exit 1.
- **Strategy**: keep `cli.py` thin — schema and persistence live in
  `persistence/`; the command wires them. Avoid: any code path that echoes the
  unmasked key (INV-001), business logic in the CLI layer.
- **Tests**: `tests/test_cli.py` scenarios in *Test Plan*.

## Constraints

- New runtime dependencies limited to `typer` (approved with this spec).
- Both store files remain human-readable JSON (`indent=2`).
- `pyright` strict and `ruff` clean per `make check`; Python 3.12.
- The persistence layer must not import Typer, read env vars (except
  `paths.py`), or print; the CLI must not touch `json` or the filesystem
  directly.
- Callers import domain stores from the domain package
  (`from resume_roast.persistence.credentials_store import CredentialsStore`),
  never from its submodules.
- Domain packages must not import each other; code shared between domains
  lives only at the persistence root (`errors.py`, `json_file.py`). This is
  what keeps encrypting `credentials_store/` later a single-domain change.
